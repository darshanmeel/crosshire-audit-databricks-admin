How to read this tour

Each entry below takes one .sql file from the library and unpacks it the same way: the system tables it reads, why it matters to an account admin, what it does in plain terms, how to read every output column, an illustrative sample of the result, and the caveats that decide whether the number can be trusted. Copy the SQL, paste it into a Databricks SQL editor, set your look-back window on the :period_days parameter, and run.

Two conventions carry through the whole library. Every dollar figure is est · at list — priced from system.billing.list_prices, pre-discount and DBU-only — unless a query explicitly joins account_prices. And usage_quantity is DBUs, bytes, hours or tokens — never dollars; only a price join turns it into money.

01
Cost & Billing cost_dollarized_by_sku_day

Dollarizing DBUs at List Price

Net DBU (and other-unit) consumption multiplied by the point-in-time list rate to produce a pre-discount list-dollar figure (net_list_cost) per usage_date x cloud x SKU x product x usage_type x unit x currency.

system.billing.usagesystem.billing.list_prices
queries/cost/cost_dollarized_by_sku_day.sql View source ↗ 
/* databricks_audit:cost_dollarized_by_sku_day */
-- NEEDS CONFIRMATION: list_rate path pricing.effective_list.default is UNVERIFIED.
-- Until confirmed, dollarize in-engine off pricing_list_prices_raw (raw JSON) instead.
SELECT u.usage_date, u.cloud, u.sku_name, u.billing_origin_product, u.usage_type, u.usage_unit,
       lp.currency_code,
       SUM(u.usage_quantity)               AS net_usage_quantity,
       SUM(u.usage_quantity * lp.list_rate) AS net_list_cost
FROM system.billing.usage u
LEFT JOIN (
  SELECT sku_name, cloud, currency_code, usage_unit, price_start_time, price_end_time,
         CAST(pricing.effective_list.default AS DOUBLE) AS list_rate   -- <-- UNVERIFIED path
  FROM system.billing.list_prices
) lp
  ON u.sku_name = lp.sku_name
 AND u.cloud    = lp.cloud
 AND u.usage_end_time >= lp.price_start_time
 AND (lp.price_end_time IS NULL OR u.usage_end_time < lp.price_end_time)
WHERE u.usage_date >= dateadd(day, -:period_days, current_date())
  AND u.usage_date < current_date()
GROUP BY u.usage_date, u.cloud, u.sku_name, u.billing_origin_product, u.usage_type, u.usage_unit, lp.currency_code
Why it matters

This is the query that turns raw DBU consumption into money, so it drives every headline cost chart, dollar-denominated anomaly series, and chargeback rollup an admin builds. The stakes are twofold: get the number and get the caveat right. net_list_cost is list (pre-discount) price only, DBU-only, and excludes cloud infra/egress, so it overstates what the account actually pays; worse, the list_rate path (pricing.effective_list.default cast to DOUBLE) is unverified on-workspace, so a silently-wrong cast could mis-scale every dollar figure downstream. Used carefully it prioritizes which SKUs and product lines to attack; used blindly it produces a confidently wrong bill.

What it does
  • Adds up all your metered usage — DBUs, plus any storage bytes, hours or tokens — for every day in the look-back window.
  • Prices each day's usage at the list rate that was in force that day, turning raw DBUs into list dollars.
  • Breaks the money down by day, cloud, SKU, product line and unit, so you can see which SKUs and product lines drive the bill and how it moves day to day.
  • Where a usage row has no matching price, it leaves the dollar figure blank rather than guessing — an unpriced gap shows up instead of a wrong number.

System tables it reads
TableRole in this query
system.billing.usageThe billed-consumption fact table (aliased u); provides usage_date, cloud, sku_name, billing_origin_product, usage_type, usage_unit, usage_end_time, and usage_quantity that is SUMmed across all record_types into net_usage_quantity and dollarized. Also carries the date-window filter.
system.billing.list_pricesThe public list (pre-discount) price history (aliased lp in a subquery); supplies currency_code and list_rate (CAST of pricing.effective_list.default to DOUBLE), joined by sku_name + cloud within the [price_start_time, price_end_time) window to price each usage row. Caveats & gotchas
  • LIST price only: net_list_cost is pre-discount, DBU-only, and excludes cloud infra/egress; the engine applies the negotiated discount_factor downstream. It is never a billed headline number.
  • UNVERIFIED price path: list_rate casts pricing.effective_list.default to DOUBLE, but docs type effective_list only as an object (and pricing.default as STRING). Until confirmed on-workspace, dollarize in-engine off pricing_list_prices_raw (raw JSON) instead of trusting net_list_cost.
  • DBUs are not dollars: net_usage_quantity is DBUs/bytes/hours/tokens in usage_unit, NOT money. Only net_list_cost is a dollar figure, and only a LIST/unverified one.
  • Point-in-time price predicate: the join uses (price_end_time IS NULL OR usage_end_time < price_end_time). price_end_time NULL means currently-active; dropping the NULL branch silently zeroes recent usage.
  • Net across corrections: SUM(usage_quantity) spans all record_type values (ORIGINAL/RETRACTION/RESTATEMENT) which already net out; the query does NOT filter to ORIGINAL and you must not re-net downstream.
  • LEFT JOIN on price: rows with no matching price window get list_rate NULL, so net_list_cost is NULL for those (usage still counted in net_usage_quantity). NULL net_list_cost means 'unpriced', not $0.
  • Join keys are sku_name + cloud + time window only; currency_code comes from the price side. usage_unit is selected in the subquery but NOT used in the ON clause, so multi-cloud/multi-currency accounts rely on cloud alone to avoid fan-out or mis-pricing (README recommends keying on cloud + currency_code + usage_unit + sku_name).
  • Date window: trailing :period_days ending yesterday (usage_date >= current_date - :period_days AND < current_date). Today is excluded because billing is incomplete; treat the most recent day as provisional due to populate lag (ingestion_date is the freshness signal).
  • Availability: both tables are GA but Unity-Catalog-only, require system.billing enabled and SELECT granted. A feature the account doesn't use yields a valid empty result, not $0; a disabled/non-permissioned table raises TABLE_OR_VIEW_NOT_FOUND. Empty != zero != error.
  • billing_origin_product has no published closed enum; a blank value is 'unattributed', not zero. Do not hardcode literals as a verified set.
→
What you do next
Use net_list_cost as the pre-discount, list-price dollar view for headline cost and anomaly charts, but do not report it as billed spend: cross-check the list_rate against pricing_list_prices_raw (dollarize in-engine off the raw JSON) before trusting any figure, and remember it excludes cloud infra/egress and the account's negotiated discount.

Cost & Billing cost_by_job

Per-Job DBU Cost Ranking

Net DBU consumption and distinct run count per job per day, split by workspace, cloud, product line, and classic-vs-serverless placement.

system.billing.usage
queries/cost/cost_by_job.sql View source ↗ 
/* databricks_audit:cost_by_job */
SELECT usage_date, cloud, workspace_id, billing_origin_product,
       usage_metadata.job_id          AS job_id,
       product_features.is_serverless AS is_serverless,
       SUM(usage_quantity) AS net_usage_quantity,
       COUNT(DISTINCT usage_metadata.job_run_id) AS distinct_runs
FROM system.billing.usage
WHERE usage_date >= dateadd(day, -:period_days, current_date())
  AND usage_date < current_date()
  AND usage_unit = 'DBU'
  AND usage_metadata.job_id IS NOT NULL
GROUP BY usage_date, cloud, workspace_id, billing_origin_product,
         usage_metadata.job_id, product_features.is_serverless
Why it matters

Jobs compute is usually the largest single product line on a Databricks bill, and this query pinpoints exactly which job_id drives the DBUs so an admin can target the worst offenders instead of the whole account. The is_serverless split lets you dollarize the jobs-on-all-purpose / classic-vs-serverless placement premium -- the same workload can cost materially more depending on where it runs. Pairing net_usage_quantity with distinct_runs surfaces failed-run and over-scheduled waste (many runs burning DBUs with little to show), which is often pure recoverable spend once the DBUs are priced.

What it does
  • Adds up the DBUs each job burned, day by day, over the look-back window.
  • Counts only jobs-compute usage — interactive notebook and SQL-editor lines are left out — so this is scheduled-job spend, nothing else.
  • Splits every job's usage by workspace and by classic-vs-serverless placement, and counts how many runs produced it.
  • Hands you the raw per-job totals to sort — the biggest spenders, and jobs racking up DBUs over many runs, rise to the top.

Cost & Billing cost_actual_vs_list_by_sku

Negotiated vs List Discount Realization

Per-SKU net DBUs alongside their negotiated-rate dollars and list-rate dollars, so an admin can quantify how much the account's negotiated discount actually saves on each SKU.

system.billing.usagesystem.billing.account_pricessystem.billing.list_prices
queries/cost/cost_actual_vs_list_by_sku.sql View source ↗ 
/* databricks_audit:cost_actual_vs_list_by_sku */
SELECT u.cloud, u.sku_name, u.usage_unit, u.billing_origin_product, u.currency_code,
       SUM(u.usage_quantity)                       AS net_usage_quantity,
       SUM(u.usage_quantity * ap.negotiated_rate)  AS net_negotiated_cost,
       SUM(u.usage_quantity * lp.list_rate)        AS net_list_cost
FROM system.billing.usage u
LEFT JOIN (
  SELECT sku_name, cloud, currency_code, usage_unit, price_start_time, price_end_time,
         CAST(pricing.default AS DOUBLE) AS negotiated_rate   -- account_prices: pricing.default ONLY (no effective_list)
  FROM system.billing.account_prices
) ap
  ON  u.sku_name      = ap.sku_name
  AND u.cloud         = ap.cloud
  AND u.currency_code = ap.currency_code
  AND u.usage_unit    = ap.usage_unit
  AND u.usage_date    >= DATE(ap.price_start_time)
  AND (ap.price_end_time IS NULL OR u.usage_date < DATE(ap.price_end_time))   -- active rows carry NULL end_time
LEFT JOIN (
  SELECT sku_name, cloud, currency_code, usage_unit, price_start_time, price_end_time,
         CAST(pricing.effective_list.default AS DOUBLE) AS list_rate   -- list_prices: effective_list.default
  FROM system.billing.list_prices
) lp
  ON  u.sku_name      = lp.sku_name
  AND u.cloud         = lp.cloud
  AND u.currency_code = lp.currency_code
  AND u.usage_unit    = lp.usage_unit
  AND u.usage_date    >= DATE(lp.price_start_time)
  AND (lp.price_end_time IS NULL OR u.usage_date < DATE(lp.price_end_time))   -- same active-row predicate
WHERE u.usage_date >= dateadd(day, -:period_days, current_date())
  AND u.usage_date < current_date()
  AND upper(u.usage_unit) = 'DBU'   -- price only DBU rows against a per-DBU rate; never blend bytes/hours/tokens
GROUP BY u.cloud, u.sku_name, u.usage_unit, u.billing_origin_product, u.currency_code
Why it matters

This is the proof that the account's negotiated Databricks contract is actually being realized: for every DBU-billed SKU it lines up what the account pays at its negotiated rate against public list, so the per-SKU gap (net_list_cost − net_negotiated_cost) is the dollar saving the discount delivers and feeds the discount-realization % and savings headline on the Pricing and Allocation tab. A SKU where the two are equal — or where net_negotiated_cost is null while DBUs are large — flags a SKU the negotiated agreement is not covering, which is real leakage against the contract. Because both dollar bases rely on unverified STRING-to-DOUBLE casts, every figure is labelled negotiated/list-unverified and must not be promoted to a billed headline until the casts are confirmed on-workspace.

What it does
  • For each SKU, totals your DBUs and prices them two ways — at your account's negotiated rate and at public list.
  • The gap between the two is the dollar saving your contract actually delivers on that SKU.
  • A SKU with usage but no negotiated price shows up as a coverage gap (blank), not as zero — so leakage against the contract stays visible.

Cost & Billing cost_chargeback_by_tag

Chargeback Coverage by Tag

Net DBU consumption and record count broken down by every custom_tags key/value pair per usage_date, cloud, workspace, and product line — with untagged usage retained as NULL-key rows so the "% untagged" denominator is real.

system.billing.usage
queries/cost/cost_chargeback_by_tag.sql View source ↗ 
/* databricks_audit:cost_chargeback_by_tag */
SELECT usage_date, cloud, workspace_id, billing_origin_product, tag_key, tag_value,
       SUM(usage_quantity) AS net_usage_quantity,
       COUNT(*) AS record_count
FROM system.billing.usage
     LATERAL VIEW OUTER explode(custom_tags) t AS tag_key, tag_value
WHERE usage_date >= dateadd(day, -:period_days, current_date())
  AND usage_date < current_date()
GROUP BY usage_date, cloud, workspace_id, billing_origin_product, tag_key, tag_value
Why it matters

Chargeback and showback only work if spend is attributable to a team, project, or cost center via tags — and this query is how an admin proves whether that attribution actually exists. The NULL-key rows produced by the OUTER explode quantify the DBUs that carry no tag at all (the % untagged spend that no one can be billed for), while counting distinct tag_values per tag_key exposes the trap where a single umbrella tag covers ~100% of DBUs yet resolves to one cost center, giving zero real team-level chargeback. On an account spending six or seven figures, untagged or falsely-"100%-tagged" DBUs are money that lands in a shared bucket and gets cross-subsidized instead of billed back to the workload that caused it.

What it does
  • Breaks your DBU usage down by every cost-allocation tag (key and value) attached to it.
  • Deliberately keeps untagged usage as its own bucket, so you get an honest '% of spend nobody can be billed for' number.
  • Cuts the tag rollup per workspace and product line — enough to spot tag drift, and to catch an umbrella tag that 'covers everything' yet names a single cost centre.

Failing Jobs Ranked by Wasted DBUs

/* databricks_audit:lakeflow_failed_jobs_wasted_dbus */
WITH end_rows AS (
  SELECT workspace_id, job_id, run_id, result_state, termination_code, period_start_time
  FROM system.lakeflow.job_run_timeline
  WHERE period_start_time >= dateadd(day, -:period_days, current_date())
    AND period_end_time < date_trunc('DAY', current_timestamp())   -- drop incomplete current day
    AND result_state IS NOT NULL                                   -- end row only
),
job_runs AS (
  SELECT workspace_id, job_id,
         COUNT(DISTINCT run_id) AS distinct_runs,
         COUNT(DISTINCT CASE WHEN result_state IN ('FAILED','ERROR','TIMED_OUT')
                             THEN run_id END) AS failed_runs
  FROM end_rows
  GROUP BY workspace_id, job_id
),
-- ARGMAX, not MAX: the termination_code of the LATEST failed run, per (workspace_id, job_id).
-- MAX(termination_code) would pick the lexicographically-largest code, not the most recent.
last_fail AS (
  SELECT workspace_id, job_id, termination_code AS last_failed_termination_code
  FROM end_rows
  WHERE result_state IN ('FAILED','ERROR','TIMED_OUT')
  QUALIFY ROW_NUMBER() OVER (
    PARTITION BY workspace_id, job_id ORDER BY period_start_time DESC
  ) = 1
),
-- DBUs billed to each job over the window. usage_metadata.job_id is unique only within a
-- workspace, so we attribute on (workspace_id, job_id). Net across ALL record_types; DBU only.
job_dbus AS (
  SELECT workspace_id,
         usage_metadata.job_id AS job_id,
         SUM(usage_quantity)   AS net_usage_quantity
  FROM system.billing.usage
  WHERE usage_date >= dateadd(day, -:period_days, current_date())
    AND usage_date < current_date()
    AND upper(usage_unit) = 'DBU'
    AND usage_metadata.job_id IS NOT NULL
  GROUP BY workspace_id, usage_metadata.job_id
)
SELECT r.workspace_id,
       r.job_id,
       r.distinct_runs,
       r.failed_runs,
       lf.last_failed_termination_code,
       COALESCE(d.net_usage_quantity, 0)                                 AS net_job_dbus,
       -- Wasted-DBU proxy: job DBUs scaled by the share of runs that failed. Honest proxy,
       -- not exact run-level waste (usage_metadata has no run_id) — labelled as such downstream.
       CASE WHEN r.distinct_runs > 0
            THEN COALESCE(d.net_usage_quantity, 0) * (r.failed_runs / r.distinct_runs)
            ELSE 0 END                                                   AS wasted_dbus_proxy
FROM job_runs r

  ON r.workspace_id = lf.workspace_id AND r.job_id = lf.job_id
LEFT JOIN job_dbus d
  ON r.workspace_id = d.workspace_id AND r.job_id = d.job_id

Warehouse Idle-Tail Seconds

Per SQL warehouse, the total seconds spent RUNNING (the auto-stop idle tail), STARTING (cold-start tax) and STOPPED, the worst single RUNNING gap, and event counts by state — all derived from state-transition events over the trailing :period_days window.

system.compute.warehouse_events
queries/compute/compute_warehouse_idle_gaps.sql View source ↗ 
/* databricks_audit:compute_warehouse_idle_gaps */
WITH ordered AS (
    SELECT
        warehouse_id,
        event_type,
        event_time,
        LEAD(event_time) OVER (PARTITION BY warehouse_id ORDER BY event_time) AS next_event_time
    FROM system.compute.warehouse_events
    WHERE event_time >= current_timestamp() - INTERVAL :period_days DAYS
),
gaps AS (
    SELECT
        warehouse_id,
        event_type,
        -- gap (seconds) this warehouse spent in `event_type` until its NEXT event.
        -- The last event per warehouse has next_event_time = NULL -> gap NULL (open, excluded).
        CASE
            WHEN next_event_time IS NULL THEN NULL
            ELSE unix_timestamp(next_event_time) - unix_timestamp(event_time)
        END AS gap_seconds
    FROM ordered
)
SELECT
    warehouse_id,
    COUNT(*)                                                                       AS event_count,
    -- seconds the warehouse held RUNNING before the next event (the auto-stop idle tail lives here)
    SUM(CASE WHEN event_type = 'RUNNING'  AND gap_seconds IS NOT NULL THEN gap_seconds ELSE 0 END) AS running_seconds,
    -- seconds spent STARTING (cold-start tax) and STOPPED (suspended, not billing compute)
    SUM(CASE WHEN event_type = 'STARTING' AND gap_seconds IS NOT NULL THEN gap_seconds ELSE 0 END) AS starting_seconds,
    SUM(CASE WHEN event_type = 'STOPPED'  AND gap_seconds IS NOT NULL THEN gap_seconds ELSE 0 END) AS stopped_seconds,
    -- longest single RUNNING gap = the worst observed idle tail before a stop/next event
    MAX(CASE WHEN event_type = 'RUNNING'  AND gap_seconds IS NOT NULL THEN gap_seconds END)         AS max_running_gap_seconds,
    SUM(CASE WHEN event_type = 'RUNNING'  THEN 1 ELSE 0 END)                        AS running_events,
    SUM(CASE WHEN event_type = 'STARTING' THEN 1 ELSE 0 END)                        AS starting_events,
    SUM(CASE WHEN event_type = 'STOPPED'  THEN 1 ELSE 0 END)                        AS stopped_events
FROM gaps
GROUP BY warehouse_id
Why it matters

A SQL warehouse keeps billing compute for the entire stretch it sits RUNNING-but-idle before auto_stop_minutes finally suspends it — that idle tail is pure waste, repeated every time the warehouse goes quiet. This query quantifies exactly how many seconds each warehouse spent in that RUNNING tail and exposes the single worst gap, so an admin can prove a warehouse's auto_stop_minutes is set too high. It also separates STARTING seconds (the cold-start tax you pay for aggressive auto-stop) so the trade-off is visible, not guessed. The numbers are seconds and event counts only; the dollar impact appears after joining warehouse DBU usage from the billing domain.

What it does
  • For each SQL warehouse, measures how long it sat RUNNING-but-idle before auto-stop finally suspended it — the idle tail you keep paying for.
  • Also separates cold-start (STARTING) time and time correctly STOPPED, and flags the single worst idle stretch.
  • Everything is in seconds and event counts, not dollars — pair it with billing to price the waste, then lower auto_stop_minutes on the offenders.

Caveats & gotchas
  • DBUs-are-not-dollars does not even apply here: system.compute.warehouse_events carries NO DBU and NO cost. Every output is in seconds or event counts — behavioral signal only. Dollarizing an idle tail needs a separate join to system.billing.usage / list prices (billing domain).
  • Date window: only events with event_time >= current_timestamp() - INTERVAL :period_days DAYS are scanned. warehouse_events retention is limited; the query assumes events exist in the window. A shorter observed history means smaller *_seconds, not a quieter warehouse.
  • Open trailing interval: the LAST event per warehouse has next_event_time = NULL, so its gap is NULL and is EXCLUDED from every SUM/MAX (never assumed zero, never 'still running forever'). event_count still counts that final event, so event_count can exceed the number of gaps that fed the seconds totals.
  • Gap is attributed to the state that OPENED it, computed with LEAD(event_time) over ALL events for the warehouse ordered by event_time — NOT after filtering to one event_type. This measures a true RUNNING->STOPPED idle tail, not the spacing between RUNNING events.
  • Enum discipline: only the authoritative 6-value enum (SCALED_UP, SCALED_DOWN, STOPPING, RUNNING, STARTING, STOPPED) is trusted; undocumented SCALING_UP/SCALING_DOWN are ignored. Only RUNNING, STARTING and STOPPED gaps are summed — SCALED_UP/SCALED_DOWN/STOPPING gap seconds are not surfaced as their own columns.
  • empty != zero != error: an empty result means no warehouse events in the window (or SQL warehouses simply not used), not idle-free warehouses. A warehouse with activity but zero RUNNING events legitimately shows running_seconds=0 / running_events=0 / max_running_gap_seconds=NULL.
  • Must-not-do: per-event query activity is deliberately NOT joined here — that join fans out at day grain and overstates query counts. This query is event-only (note: the warehouse-idle finding PAGE separately LEFT JOINs the full warehouse list to surface unused warehouses, but this specific query does NOT — an unused warehouse with no events simply produces no row).
  • Availability: system.compute.warehouse_events is GA but requires Unity Catalog, the compute schema enabled on the metastore, and SELECT on system.*. Tables are regional — run per metastore region and union if needed.
  • No masking in this query: warehouse_id is a UUID passed through as-is (no warehouse_name is selected, so no name truncation applies).

What you do next

A warehouse showing max_running_gap_seconds far larger than its configured auto_stop_minutes*60 is either mis-configured or being kept warm by trickle queries — investigate before tuning.

Your Heaviest Statements, Ranked

The top 1000 finished, non-cached SQL statements in the trailing window, ranked by execution time (a DBU-cost proxy), each carrying pruning, spill, shuffle, and scan counters plus a de-valued query text to diagnose the fix.

queries/performance/query_costly_statements.sql View source ↗ 
/* databricks_audit:query_costly_statements */
SELECT
  workspace_id,
  statement_id,
  statement_type,
  -- executed_by: partial-unmask (house style) — keep an email as da****@****, keep a service-principal
  -- GUID as-is (opaque handle), else first-2-chars + ****. No raw username/email reaches the CSV.
  CASE
    WHEN executed_by IS NULL OR executed_by = '__REDACTED__' THEN executed_by
    WHEN executed_by LIKE '%@%' THEN concat(substr(executed_by, 1, 2), '****@****')
    WHEN executed_by RLIKE '^[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{12}$' THEN executed_by
    ELSE concat(substr(executed_by, 1, 2), '****')
  END                                    AS executed_by,
  compute.warehouse_id                   AS warehouse_id,
  compute.type                           AS compute_type,
  start_time,
  execution_duration_ms,
  waiting_for_compute_duration_ms,
  total_task_duration_ms,
  read_bytes,
  read_files,
  pruned_files,
  read_partitions,
  read_rows,
  produced_rows,
  spilled_local_bytes,
  shuffle_read_bytes,
  from_result_cache,
  -- statement_text de-valued at source: strip emails, then replace every single-quoted string literal
  -- with '?' (chr(39) is the single quote, avoiding SQL-string escaping). Keeps the query shape
  -- (table/column identifiers, structure) while removing literal data values.
  regexp_replace(
    regexp_replace(statement_text, '[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+[.][A-Za-z]{2,}', '<email>'),
    concat(chr(39), '[^', chr(39), ']*', chr(39)), '?'
  )                                      AS statement_text
FROM system.query.history
WHERE start_time >= dateadd(day, -:period_days, current_date())
  AND start_time <  current_date()
  AND execution_status = 'FINISHED'
  AND from_result_cache = false
  AND execution_duration_ms > 0
ORDER BY execution_duration_ms DESC
LIMIT 1000
Why it matters

This is the single best 'what to tune first' worklist: Databricks exposes no per-query dollar column, so ranking by execution_duration_ms is the closest proxy for DBU spend, and the top of this list is where money is actually burning. The attached counters tell you the fix rather than just the symptom: poor pruning (full scans) points at partitioning/clustering, spilled_local_bytes points at an undersized warehouse or memory-heavy rewrite, and large shuffle_read_bytes flags a bad join. Fixing the handful of statements at the top typically moves the warehouse's bill more than any autoscaling knob.

What it does
  • Lists your heaviest finished SQL statements, ranked by execution time — the closest proxy Databricks gives for per-query cost.
  • Attaches the diagnostic counters for each — pruning, spill, shuffle, scan volume — so you can see why it's heavy, not just that it is.
  • Strips data values out of the query text (emails and literals), leaving only the query shape, so nothing sensitive leaves the workspace.
  • Caps at the top 1000 by time, so a light-but-frequent query won't necessarily appear here.

Governance, Access & Security access_runas_escalation

Run-As Privilege Escalation Watch

A 30-day rollup of audited actions where the initiating principal (run_by) differs from the identity the action executed as (run_as), aggregated per service, action, and masked identity pair with event counts and first/last timestamps.

system.access.audit
queries/governance_access/access_runas_escalation.sql View source ↗ 
/* databricks_audit:access_runas_escalation */
SELECT service_name, action_name,
       CASE
         WHEN identity_metadata.run_by IS NULL OR identity_metadata.run_by = '__REDACTED__' THEN identity_metadata.run_by
         WHEN identity_metadata.run_by LIKE '%@%' THEN concat(substr(identity_metadata.run_by, 1, 2), '****@****')
         WHEN identity_metadata.run_by RLIKE '^[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{12}$' THEN identity_metadata.run_by
         ELSE concat(substr(identity_metadata.run_by, 1, 2), '****')
       END AS run_by,
       CASE
         WHEN identity_metadata.run_as IS NULL OR identity_metadata.run_as = '__REDACTED__' THEN identity_metadata.run_as
         WHEN identity_metadata.run_as LIKE '%@%' THEN concat(substr(identity_metadata.run_as, 1, 2), '****@****')
         WHEN identity_metadata.run_as RLIKE '^[0-9a-fA-F]{8}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{4}-[0-9a-fA-F]{12}$' THEN identity_metadata.run_as
         ELSE concat(substr(identity_metadata.run_as, 1, 2), '****')
       END AS run_as,
       COUNT(*) AS event_count,
       MIN(event_time) AS first_event_time, MAX(event_time) AS last_event_time
FROM system.access.audit
WHERE identity_metadata.run_by IS NOT NULL
  AND identity_metadata.run_as IS NOT NULL
  AND identity_metadata.run_by <> identity_metadata.run_as
  AND event_date >= current_date() - INTERVAL 30 DAYS
  AND event_date < current_date()
GROUP BY 1, 2, identity_metadata.run_by, identity_metadata.run_as
Why it matters

Run-as escalation is the audit fingerprint of one principal executing under another identity's authority — the exact move behind privilege abuse, lateral movement, or a compromised service principal quietly acting as a human owner. Surfacing every run_by <> run_as pair with counts and recency lets an admin separate legitimate job/service-principal execution from unauthorized impersonation before it turns into data exfiltration or unreviewed permission changes. Because the signal is genuinely sparse (identity_metadata is usually NULL for ordinary actions), each row that does appear is worth an explicit look, and an empty result must be verified rather than trusted as clean.

What it does
  • Surfaces every audited action where the identity that triggered it differs from the identity it ran as — the fingerprint of run-as escalation.
  • Rolls each run-by → run-as pair up with how many times it happened and when it was first and last seen.
  • Masks the identities, and — because the signal is naturally rare — treats an empty result as 'check the audit log is on', not 'all clear'.
